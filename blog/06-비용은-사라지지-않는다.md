# 비용은 사라지지 않는다 — 읽기에서 치를 것인가, 쓰기에서 치를 것인가

## 핵심 질문

> 성능을 "개선"한다고 말하지만, 실제로는 비용을 **읽기에서 쓰기로 옮긴 것**뿐인 경우가 대부분이다. 비용은 사라지는가, 이동하는가?

## 다루고 싶은 것

- DB ORDER BY는 읽기 시점에 정렬 비용을 치르고, Redis ZSET은 쓰기 시점에 O(logN)으로 미리 치른다
- 이것은 ZSET만의 이야기가 아니라 소프트웨어 전반에서 반복되는 패턴
  - 캐시: 읽기 비용을 쓰기(캐시 적재) 시점으로 이동
  - 인덱스: 조회 비용을 INSERT/UPDATE 시점으로 이동
  - 선계산(Pre-computation): 요청 시점의 연산을 배치/이벤트 시점으로 이동
  - CQRS: 읽기 모델과 쓰기 모델을 분리하여 각각 최적화
- "어디서 비용을 치를 것인가"를 결정하는 기준은 읽기/쓰기 비율
  - 랭킹 API: 읽기가 압도적 (1:99 ~ 1:9) → 쓰기 시점에 비용을 치르는 것이 유리
  - 로그 적재: 쓰기가 압도적 → 읽기 시점에 비용을 치르는 것이 유리 (쓰기 최적화)
- 비용 이동이 아닌 진짜 "제거"처럼 보이는 경우도 결국 다른 형태의 비용을 수반
  - 캐시 → 정합성 비용 (Cache Stampede, 만료 전략)
  - 인덱스 → 저장 공간 + 쓰기 성능 비용
  - ZSET → 메모리 비용 + 단일 score 제약
- DB에 랭킹 부하가 집중되면 주문/상품 등 핵심 비즈니스 쿼리에도 영향 → "부하 격리"의 본질

## 출발점

- Redis ZSET으로 랭킹을 구현하면서 "왜 DB ORDER BY가 아닌가?"라는 질문에서 시작
- "ZSET이 빠르다"가 아니라, "비용을 치르는 시점이 다르다"가 진짜 답

# "거부"와 "대기"는 설계 철학이 다르다 — Rate Limiting vs Queuing의 본질

## 들어가며

블랙 프라이데이 행사가 시작된다. 평소 초당 100건이던 주문 요청이 10,000건으로 폭증한다. 시스템이 감당할 수 있는 건 초당 200건 정도. 나머지 9,800건을 어떻게 할 것인가?

이 질문에 대한 답은 크게 두 가지다.

1. **"처리할 수 없으니 거부합니다"** — 429 Too Many Requests
2. **"처리할 수 없으니 기다려주세요"** — 대기열 진입, 현재 512번째

같은 과부하, 같은 기술적 한계. 하지만 시스템이 유저에게 건네는 메시지는 완전히 다르다. 이 차이는 기술 선택이 아니다. **초과 트래픽을 적으로 볼 것인가, 고객으로 볼 것인가** — 관점의 차이다.

이 글에서는 세 가지를 다룬다.

1. Rate Limiting과 Queuing이 동일한 문제에 대해 정반대의 메시지를 보내는 이유
2. 각각이 적합한 상황의 경계
3. 양자택일이 아니라 조합할 수 있다는 점

---

## 같은 과부하, 다른 메시지

### 429 — "다시 오세요"

```
유저: POST /orders
서버: 429 Too Many Requests
      Retry-After: 30

유저 해석: "지금은 안 된다. 나중에 다시 해봐야지."
```

Rate Limiting은 **초과 요청을 잘라내는 것**이다. Token Bucket이든 Sliding Window든, 한계를 넘은 요청에 대해 시스템이 하는 일은 하나다. 거부. 유저에게 전달되는 메시지는 "나중에 다시 시도하세요"이고, 언제 되는지는 알려주지 않는다.

이 방식에는 명확한 장점이 있다. **시스템이 즉시 보호된다.** 요청을 거부하는 데 드는 비용은 거의 0이다. 커넥션을 유지할 필요도 없고, 상태를 저장할 필요도 없다. 서버 입장에서는 가장 경제적인 방어 수단이다.

### 대기열 — "기다려주세요, 512번째입니다"

```
유저: POST /queue/enter
서버: 200 OK
      { "position": 512, "estimatedWait": "약 3분" }

유저 해석: "512번째구나. 3분이면 기다릴 만하네."
```

Queuing은 **초과 요청을 보관하는 것**이다. 거부하지 않는다. 대신 순서를 부여하고, 시스템이 감당할 수 있는 속도로 하나씩 처리한다. 유저에게 전달되는 메시지는 "잠시만 기다려주세요"이고, 내 순번과 예상 시간이 함께 온다.

대신 비용이 따른다. 대기열을 저장할 인프라(Redis 등)가 필요하고, 순번 조회 요청이 계속 들어오고, 이탈한 유저의 자리를 회수하는 로직도 필요하다. **유저를 기다리게 하는 것은 거부하는 것보다 훨씬 복잡하다.**

### 본질적 차이

두 방식의 차이를 한 문장으로 정리하면 이렇다.

> **Rate Limiting은 초과 트래픽을 버린다. Queuing은 초과 트래픽을 보관한다.**

| | Rate Limiting | Queuing |
|---|---|---|
| 초과 요청에 대한 태도 | 거부 (429) | 보관 (대기열 진입) |
| 유저에게 전달하는 메시지 | "나중에 다시 시도하세요" | "512번째입니다, 약 3분" |
| 시스템 비용 | 거의 0 | 인프라 + 상태 관리 필요 |
| 유저의 다음 행동 | 재시도 (언제 될지 모름) | 대기 (순번이 보임) |
| 처리 보장 | 없음 | 있음 (순서대로 처리) |

기술적으로는 둘 다 "시스템이 감당할 수 있는 양만 처리한다"는 같은 목적을 달성한다. 하지만 **유저가 경험하는 것**은 완전히 다르다. 거부당하는 것과 기다리는 것은 같지 않다.

## 429가 만드는 악순환 — 재시도 폭풍

Rate Limiting의 가장 위험한 부작용은 **유저의 재시도**다.

```
t=0s   10,000명 동시 접속. 200명 통과, 9,800명 거부 (429)
t=5s   거부당한 9,800명 중 60%가 재시도 → 5,880명 + 신규 유입
       → 실제 요청: ~12,000건
t=10s  다시 거부 → 또 재시도 → 요청 더 증가
       → 실제 요청: ~15,000건
...
→ 처리량은 그대로 200 TPS인데, 요청은 계속 불어난다
```

429를 받은 유저는 브라우저를 새로고침하거나, 뒤로 가서 다시 주문 버튼을 누른다. 이게 한 명이면 무해하지만, **9,800명이 동시에** 하면 상황이 달라진다. 거부된 요청이 재시도로 돌아와 원래보다 더 큰 부하를 만든다. 이것이 **재시도 폭풍(Retry Storm)**이다.

```
[최초 요청]
    ↓
[Rate Limiting: 429 반환]
    ↓
[유저 재시도] ──→ [Rate Limiting: 또 429]
    ↓                    ↓
[또 재시도]          [또 재시도]
    ↓                    ↓
[요청 총량 증가] ←────────┘
    ↓
[서버 부하 유지 or 증가]
```

물론 `Retry-After` 헤더로 재시도 간격을 지정할 수 있고, Exponential Backoff를 적용할 수도 있다. 하지만 이건 **클라이언트가 협조할 때만** 작동한다. 브라우저에서 유저가 직접 새로고침을 누르는 행위는 어떤 서버 설정으로도 막을 수 없다.

대기열은 이 문제를 구조적으로 해결한다. 유저가 대기열에 진입한 순간, **더 이상 재시도할 이유가 없어진다.** 순번이 보이고, 줄어드는 것이 보이기 때문이다. "기다리면 된다"는 확신이 재시도 충동을 없앤다.

> Rate Limiting은 유저를 밀어내고, Queuing은 유저를 잡아둔다. 재시도 폭풍은 밀어낸 대가다.

## 경계 — 언제 거부하고, 언제 기다리게 할 것인가

그렇다면 대기열이 항상 정답인가? 아니다. Rate Limiting이 더 적합한 상황이 분명히 존재한다.

### Rate Limiting이 맞는 상황

**1. API 보호 — 호출자가 기다릴 필요가 없는 경우**

```
외부 파트너 API 호출
→ 초당 1,000건 제한
→ 1,001번째 요청: 429 Too Many Requests

이 호출자에게 "대기열에서 기다리세요"는 의미가 없다.
클라이언트가 알아서 재시도 로직을 구현해야 하는 영역이다.
```

API를 소비하는 쪽이 사람이 아니라 **다른 시스템**이면, 대기열은 과도하다. 시스템은 429를 받고 Exponential Backoff로 재시도하면 된다. 감정도 없고, 이탈도 없다.

**2. 봇/어뷰징 차단 — 애초에 처리할 의사가 없는 요청**

```
특정 IP에서 초당 500건의 요청
→ 이건 유저가 아니라 봇이다
→ 429로 거부하거나 아예 차단해야 한다

이 요청을 대기열에 넣으면?
→ 봇이 대기열의 자리를 차지해서 정상 유저가 밀린다
```

악의적인 트래픽을 대기열에 넣는 것은 오히려 해롭다. **처리할 의사가 없는 요청**은 거부가 맞다.

### Queuing이 맞는 상황

**유저가 기다릴 의사가 있고, 처리가 보장되어야 하는 경우**

```
블랙 프라이데이 한정 상품 주문
→ 유저는 이 상품을 사려고 기다리고 있다
→ "나중에 다시 시도하세요"를 받으면? 경쟁사로 간다.
→ "512번째입니다, 약 3분"을 받으면? 기다린다.
```

핵심은 **유저의 기다림에 가치가 있는가**다. 유저가 기꺼이 기다릴 만한 목적이 있고, 기다리면 반드시 처리해줄 수 있을 때 — 대기열이 의미를 가진다.

기다릴 의사가 없는 유저에게 대기열을 보여주는 것은 오히려 부정적이다. 뉴스 기사를 읽으려는데 "현재 3,000번째입니다"가 뜨면? 그냥 다른 곳으로 간다.

### 판단 기준

| 조건 | Rate Limiting | Queuing |
|------|:---:|:---:|
| 호출자가 시스템(API to API)인가? | O | |
| 악의적 트래픽을 걸러내야 하는가? | O | |
| 유저가 기다릴 의사가 있는가? | | O |
| 처리를 보장해야 하는가? | | O |
| 요청의 순서가 중요한가? | | O |

가장 중요한 질문 하나로 줄이면 이렇다.

> **이 요청의 주인이 "기다려서라도 결과를 얻고 싶은 사람"인가?**

그렇다면 대기열. 아니라면 Rate Limiting.

## 조합 — 거부와 대기는 같이 쓸 수 있다

실전에서는 둘 중 하나만 쓰는 경우가 드물다. **1차로 걸러내고, 2차로 줄 세우는 것**이 가장 자연스러운 구조다.

![요청 흐름 다이어그램](img/request_flow_diagram.svg)

1차 Rate Limiting은 **시스템 보호**가 목적이다. 특정 IP에서 비정상적으로 많은 요청이 오면, 이건 유저가 아니라 봇이거나 스크립트다. 이 트래픽이 대기열에 들어오면 정상 유저의 순번만 밀린다. 앞단에서 걸러내는 것이 맞다.

2차 대기열은 **유저 경험**이 목적이다. 1차를 통과한 정상 유저에게 순번을 부여하고, 시스템이 감당할 수 있는 속도로 하나씩 통과시킨다. "기다리면 반드시 처리된다"는 신뢰를 제공하는 것이다.

이 조합이 핵심적인 이유가 있다. **봇은 429로 거부하고, 정상 유저는 대기열에서 기다리게 한다.** 초과 트래픽 중에서 적과 고객을 분리하는 것이다.

---

## 마무리 — 거부와 대기는 관점의 문제다

Rate Limiting과 Queuing은 같은 문제를 해결하는 다른 기술이 아니다. **같은 상황을 바라보는 다른 관점**이다.

> **Rate Limiting은 "이 요청은 감당할 수 없다"는 판단이다**
> - 초과 요청을 잘라낸다. 비용이 적고, 시스템이 즉시 보호된다
> - 하지만 유저가 재시도하면서 부하가 오히려 늘어나는 악순환이 발생할 수 있다
> - API 보호, 봇 차단처럼 **호출자를 기다리게 할 이유가 없는 상황**에 적합하다

> **Queuing은 "이 요청은 지금은 처리할 수 없지만, 반드시 처리하겠다"는 약속이다**
> - 초과 요청을 보관하고 순서대로 처리한다. 유저에게 순번과 예상 시간을 알려준다
> - 대신 인프라 비용이 들고, 대기 중 이탈 관리라는 새로운 문제가 생긴다
> - 유저가 **기다려서라도 결과를 얻고 싶은 상황**에 적합하다

> **실전에서는 조합이다**
> - 봇은 Rate Limiting으로 거부하고, 정상 유저만 대기열에 진입시킨다
> - 초과 트래픽 중에서 적과 고객을 분리하는 것이 핵심이다

결국 설계자가 먼저 답해야 하는 질문은 기술이 아니다.

**"초과 트래픽을 적으로 볼 것인가, 고객으로 볼 것인가?"**

이 질문에 대한 답이 429를 반환할지, 순번을 부여할지를 결정한다. 그리고 그 답은 시스템 아키텍처가 아니라, 우리 서비스가 유저에게 어떤 경험을 제공하고 싶은지에서 나온다.

다음 글에서는 "적절히 느리게"라는 사고방식 — Back-pressure의 본질을 다룬다.

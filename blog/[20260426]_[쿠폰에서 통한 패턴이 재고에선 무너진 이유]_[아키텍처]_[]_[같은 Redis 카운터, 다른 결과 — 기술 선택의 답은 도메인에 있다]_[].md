# 쿠폰에서 통한 패턴이 재고에선 무너진 이유

## 들어가며

선착순 100장 한정 쿠폰 발급. 200명이 동시에 몰린다. DB 락은 처리량에서 깨진다. Redis `INCR`로 100명까지만 통과시키고 그다음은 거절한다. 잘 돌아갔다. 끝.

같은 패턴을 재고 차감에 그대로 적용했다. Redis 카운터로 미리 차감하고 DB에 INSERT. 단위 테스트도 통과했고 통합 테스트도 통과했다. 코드는 거의 같았다.

그런데 한 달 뒤, Redis 카운터와 실제 DB 재고가 어긋나기 시작했다. 보상 로직이 0.1% 확률로 실패했고, 그 누설이 매일 누적됐다. 결국 정합성을 잡기 위해 별도 배치를 돌리기 시작했다.

**같은 기술, 같은 코드, 다른 결과.** 무엇이 달랐을까?

이 글에서는 세 가지를 다룬다.

1. 쿠폰 발급에서 Redis 카운터가 안전했던 이유
2. 재고 차감에서 같은 패턴이 무너진 이유
3. 두 케이스가 갈린 지점은 코드가 아니라 도메인이라는 것

---

## 같은 패턴 — Redis 카운터 + DB

먼저 두 도메인에 적용된 코드를 보자. 거의 같다.

### 쿠폰 발급

```kotlin
@Transactional
fun issueCoupon(userId: Long, eventId: Long): Coupon {
    val count = redisTemplate.opsForValue().increment("coupon:event:$eventId")!!
    if (count > MAX_ISSUE) {
        redisTemplate.opsForValue().decrement("coupon:event:$eventId")
        throw CoreException(ErrorType.CONFLICT, "쿠폰이 모두 소진되었습니다")
    }
    return couponRepository.save(Coupon(userId, eventId))
}
```

### 재고 차감

```kotlin
@Transactional
fun deductStock(productId: Long, quantity: Int): Order {
    val remain = redisTemplate.opsForValue().decrement("stock:$productId", quantity.toLong())!!
    if (remain < 0) {
        redisTemplate.opsForValue().increment("stock:$productId", quantity.toLong())
        throw CoreException(ErrorType.CONFLICT, "재고가 부족합니다")
    }
    return orderRepository.save(Order(productId, quantity))
}
```

Redis로 사전 점유, 한도 검사, 실패 시 보상, 성공 시 DB에 기록. 패턴은 동일하다.

이 패턴은 같은 약점도 공유한다. **Redis와 DB는 한 트랜잭션에 묶이지 않는다.** 위 코드의 `@Transactional`은 DB 저장만 감싸고, Redis 호출은 네트워크 너머의 별개 시스템이다. Redis 점유는 성공했는데 DB INSERT가 네트워크 장애로 실패하면 보상을 해야 하고, 그 보상마저 실패하면 카운터가 어긋난다.

기술적 약점은 동일하다. 그런데 결과는 달랐다.

---

## 쿠폰에서는 왜 안전했나

### 비즈니스 성격: 이벤트성

쿠폰 발급은 **이벤트성**이다. 한정 100장의 쿠폰이 풀리고, 200명이 몰리고, 끝난다. 트래픽은 시작과 끝이 명확하고, 같은 이벤트는 다시 일어나지 않는다.

이 성격이 만든 결과는 두 가지다.

**1. 누설이 누적되지 않는다**

이벤트가 끝나면 카운터의 역할도 끝난다. 보상 실패로 카운터가 1 어긋났다고 해도, 다음 이벤트는 새 카운터(`coupon:event:101`)에서 시작한다. 어긋남은 *그 이벤트 한정*이고, 다음 이벤트로 전파되지 않는다.

**2. 운영 보정이 1회로 끝난다**

이벤트가 끝나고 *Redis 카운터의 발급 횟수 ≠ DB의 쿠폰 행 수*라면, 운영팀이 한 번 비교해서 누락된 만큼 보충 발급하면 된다. 그게 끝이다. 다음 주에 또 보정할 일은 없다.

### Under-issue is safe

게다가 쿠폰의 비즈니스 규칙 자체가 **under-issue를 허용**한다. 100장 중 99장만 발급되었다고 해도 비즈니스 손실은 거의 없다. 반대로 100장 한도에서 101장이 발급되면 한 명에게 회수 메일을 보내야 하고, 회수 거부 시 비즈니스 비용이 발생한다.

**99장 발급은 아무도 모른다. 101장 발급은 보상이 필요하다.**

이 비대칭 위에서 우리는 의식하지 못한 채 안전한 트레이드오프를 한 것이다.

> "보상 실패로 카운터가 어긋나도, under-issue 방향이라면 비즈니스 손실은 없다. 한 번의 이벤트 보정으로 끝낼 수 있다."

이 트레이드오프가 통한 이유는 코드가 좋아서가 아니라 **도메인이 그것을 허용**했기 때문이다.

## 재고에서는 왜 무너졌나

### 비즈니스 성격: 상시성

재고 차감은 **상시성**이다. 시스템이 살아 있는 한 계속 작동한다. 하루에 1만 건의 주문이 들어오고, 1천 건의 취소가 들어오고, 그다음 날도 똑같이 반복된다.

이 성격이 만든 결과는 두 가지다.

**1. 누설이 누적된다**

같은 0.1% 보상 실패율을 두 도메인에 적용해보자.

- 쿠폰: 200건/이벤트 × 0.1% = 평균 0.2건 누설. 한 번 보정.
- 재고: 1만 건/일 × 0.1% = 평균 10건 누설. **매일.**

한 달이면 약 300건이 어긋난다. 카운터는 시간이 갈수록 진실에서 멀어진다. 이걸 **continuous leak**이라고 부르자. 시간 축에서 누적되는 누설이다.

> 별도로, `DECR` → 음수 검출 → `INCR` 보상 사이의 짧은 창에서 다른 요청이 음수 카운터를 보고 거절하는 *phantom rejection*도 발생한다. 이 글에서는 **시간 축 누설**에 집중하지만, 동시성 race 또한 같은 패턴의 또 다른 비용임을 짚어둔다.

**2. 운영 보정을 한 번으로 끝낼 수 없다**

쿠폰처럼 "이벤트 끝나고 한 번 비교"라는 모델이 통하지 않는다. 재고는 *끝나지 않기 때문*이다. 매일 어긋나는 만큼 매일 동기화해야 하고, 그 동기화 자체가 시스템의 일부가 된다.

### 같은 0.1%, 다른 의미

같은 보상 실패율인데 한쪽은 *한 번의 사고*고 한쪽은 *지속적인 출혈*이다.

| | 쿠폰 | 재고 |
|---|---|---|
| 보상 실패율 | 0.1% | 0.1% |
| 트래픽 규모 | 200건/이벤트 | 1만 건/일 |
| 1회 누설량 | 평균 0.2건 | 평균 10건 |
| 누적성 | 없음 (이벤트 종료 시 리셋) | 있음 (시간 축으로 누적) |
| 보정 빈도 | 1회 | 주기적 (매일 또는 N분) |

**0.1%는 같은 숫자다. 하지만 비즈니스의 시간성이 그 숫자의 의미를 다르게 만든다.**

### Under-issue가 안전해도 충분하지 않다

재고에서 under-issue에 해당하는 방향은 **카운터가 실재고보다 적게 보이는 경우**다. 시스템은 9개라고 믿지만 실제로는 10개가 있는 상황. 이 방향이라면 오버셀링은 없고 "재고 부족" 메시지가 한 번 더 뜨는 정도라 즉각적인 비즈니스 손실은 미미하다.

하지만 쿠폰과 다른 점은 **그 손실이 시간이 지나면 누적되어 보인다**는 것이다. 한 번의 under-issue는 안 보이지만, 한 달간 누적된 under-issue는 *카운터와 실제 재고의 불일치*로 드러난다. 운영팀이 매일 "왜 카운터에는 50개 있는데 실제 재고는 47개입니까?"라는 질문을 받기 시작한다.

같은 트레이드오프가 쿠폰에서는 끝나지만, 재고에서는 *끝나지 않는다*.

## 같은 코드, 다른 결과를 만든 것

### 차이를 만든 것은 코드가 아니다

위의 두 케이스를 다시 보자. 코드는 거의 같았다. 사용한 기술도 같았다. Redis도 같고, DB도 같고, 보상 로직의 구조도 같았다.

다른 건 단 하나다. **비즈니스의 시간성.**

- 쿠폰은 시간 축에서 *끊긴다*. 어긋남은 시작과 끝이 있다.
- 재고는 시간 축에서 *흐른다*. 어긋남은 누적된다.

이 차이는 코드를 보고는 알 수 없다. 도메인을 봐야 알 수 있다.

### 처음 던졌어야 할 질문

"Redis 카운터로 풀 수 있나?"라는 질문은 *기술의 질문*이다. 답은 "예"다. 두 케이스 모두에서.

하지만 답이 같다고 결과도 같지 않다는 게 이번 케이스의 교훈이다. 처음 던졌어야 할 질문은 다른 것이다.

> "이 도메인은 시간이 흐를 때 무엇이 누적되는가? 어떤 오차를 허용하는가?"

이 질문을 던졌다면, 재고에서는 처음부터 *주기적 reconciliation*을 설계의 일부로 넣었을 것이다. Redis는 hot-path 카운터로만 쓰고, DB의 `stock`을 진실의 원천(source of truth)으로 두고, N분마다 또는 트래픽 한산 시간대에 DB → Redis 재계산을 돌렸을 것이다.

쿠폰에서는 굳이 그런 인프라를 만들 필요가 없었다. 한 번의 운영 보정으로 충분했으니까.

이 글에서는 우연히 *쿠폰=이벤트성, 재고=상시성*으로 매칭됐지만, 진짜 변수는 도메인의 이름이 아니라 **시간성**이다. 가입 즉시 발급되는 상시 쿠폰이라면 재고와 같은 주기적 reconciliation이 필요하고, 1시간 한정 플래시 세일 재고라면 쿠폰처럼 한 번의 보정으로 끝낼 수도 있다.

**도메인이 다르면 같은 기술이라도 다른 모습으로 배치되어야 한다.**

---

## 마치며

기술 선택은 종종 *기술 자체의 매력*에 끌려간다. "Redis가 빠르다", "락이 안전하다", "Kafka가 견고하다" 같은 말은 사실이다. 하지만 그것만으로는 어떤 도메인에 어떻게 써야 하는지 답하지 못한다.

같은 Redis 카운터가 한쪽에서는 안전하고 한쪽에서는 무너진 이유는 코드 문제가 아니었다. 도메인이 다른데 기술만 같았기 때문이다. 그리고 그 차이를 알아채기 전에는, 어떤 코드를 써도 결과는 같았을 것이다.

내가 지금 짜고 있는 시스템에 대해 던질 첫 질문은 이것이다. **카운터의 어긋남이 한 번의 사고인가, 지속적인 출혈인가?** 답이 후자라면 reconciliation은 부록이 아니라 본문이다.

기술 자체로는 답이 정해지지 않는다. 도메인이 답을 정한다.

# WIL - 인덱스와 캐시로 성능을 설계한 한 주

## 이번 주 요약

이번 주는 **인덱스 최적화**에서 시작해 **캐시 아키텍처 설계와 구현**까지 이어진 한 주였다. 인덱스를 분석하고 복합 인덱스를 설계하면서 쿼리 레벨의 성능을 확보한 뒤, Redis와 Caffeine을 조합한 2계층 캐시 아키텍처를 직접 설계하고 구현했다. "조회 성능"이라는 하나의 주제를 쿼리 최적화와 캐시 전략이라는 두 축으로 풀어낸 한 주였다. 글로 정리하고, 코드로 증명하는 순환이 이번 주에도 이어졌다.

---

## 이번 주에 정리한 것

### 멀티 레이어 캐시 전략 - Local Cache와 Redis를 함께 쓰는 이유

Redis 캐시를 도입하면 DB 부하가 줄어든다. 하지만 Redis도 결국 네트워크를 타는 외부 저장소다. 매 요청마다 ~1ms의 네트워크 왕복이 발생하고, Redis 장애 시 모든 인스턴스가 영향을 받는다.

해결책은 Redis 앞에 **Caffeine 로컬 캐시**를 하나 더 두는 것이었다. 이 글에서 다룬 핵심은 다음과 같다.

- **Cache-Aside 패턴**: 읽기 시점에 캐시가 없으면 DB에서 가져와 적재. 가장 범용적이고 안정적인 전략.
- **2계층 구조**: Local Cache(0ms, 인스턴스별) → Redis(~1ms, 공유) → DB(~5-50ms). 각 계층이 다른 역할을 수행.
- **캐시 워밍**: 서버 시작 직후 인기 데이터를 미리 적재하여 Cold Start 방지. 워밍 주기 < TTL이 핵심 조건.
- **DIP 적용**: ProductCacheRepository, ProductLocalCacheRepository를 인터페이스로 분리. Application 계층은 Caffeine이나 Redis를 알지 못한다.

캐시는 "데이터를 저장해두는 것"이 아니라, **조회 순서의 오케스트레이션을 설계하는 것**이라는 관점을 정리했다.

---

### Cache Stampede - 캐시가 만료되는 순간 벌어지는 일

캐시를 도입하면 성능이 좋아진다. 하지만 **캐시가 만료되는 바로 그 순간**, 오히려 캐시가 없을 때보다 더 심각한 장애가 발생할 수 있다. 초당 1,000건의 요청이 들어오는 API에서 캐시 TTL이 만료되면, 1,000건의 동일한 쿼리가 DB로 쏟아진다.

세 가지 해결 전략을 비교했다.

| 전략 | 원리 | 적합한 캐시 계층 |
|------|------|-----------------|
| 뮤텍스 | 하나의 스레드만 DB 조회하도록 락 | 범용적, 분산 환경에선 분산 락 필요 |
| 스케줄러 워밍 | TTL 만료 전에 주기적으로 갱신 | **Redis (공유 캐시)** |
| Caffeine 원자적 로딩 | `Cache.get(key, loader)`로 키별 자동 동기화 | **로컬 캐시** |

결론은 **캐시 계층별로 다른 전략을 적용**하는 것이 가장 효과적이라는 것이었다. 로컬 캐시에는 Caffeine 원자적 로딩, Redis에는 스케줄러 워밍. 두 전략을 조합하면 어느 계층에서든 Stampede를 방지할 수 있다.

## 이번 주의 깨달음

이번 주는 **"성능을 올리는 것"과 "성능을 설계하는 것"의 차이**를 느낀 한 주였다.

인덱스를 추가하면 빨라진다. 캐시를 도입하면 빨라진다. 여기까지는 누구나 안다. 하지만 "어떤 인덱스를", "어떤 캐시 계층에", "어떤 TTL과 무효화 전략으로" 적용할지를 결정하려면, 그 기술이 **내부에서 어떻게 동작하는지**를 이해해야 한다.

- 인덱스의 카디널리티와 조회 패턴을 모르면, 복합 인덱스의 컬럼 순서를 결정할 수 없다.
- Cache Stampede를 모르면, 캐시 TTL 만료 순간에 DB가 죽는 것을 사후에야 알게 된다.
- DIP를 적용하지 않으면, 로컬 캐시를 추가하는 순간 Application 코드 전체를 수정해야 한다.

**성능 최적화의 본질은 "빠르게 만드는 것"이 아니라 "느려질 수 있는 지점을 예측하고 설계로 방어하는 것"이다.** 인덱스 설계와 캐시 아키텍처 모두 결국 같은 질문에 답하고 있었다 — "이 시스템이 부하를 받을 때 어디서 병목이 생기고, 어떻게 대비할 것인가."

쿼리 레벨에서는 인덱스로, 애플리케이션 레벨에서는 캐시로 같은 문제를 다른 계층에서 풀었다. 결국 성능은 한 가지 기술로 해결하는 것이 아니라, **각 계층에서 적절한 전략을 조합하는 것**이다.

# 테스트가 증명하는 좋은 도메인 서비스 설계

## 들어가며

"이 코드 테스트 어떻게 짜요?"

코드 리뷰에서 가장 자주 듣는 질문 중 하나다. 그리고 이 질문이 나온다는 건, 대부분 **설계에 문제가 있다는 신호**다.

[이전 글](https://zzangoobrother.github.io/BLOG/?post=%5B20260301%5D_%5B%EC%9D%91%EC%9A%A9+%EC%84%9C%EB%B9%84%EC%8A%A4%EC%99%80+%EB%8F%84%EB%A9%94%EC%9D%B8+%EC%84%9C%EB%B9%84%EC%8A%A4+-+%EC%96%B4%EB%94%94%EC%97%90+%EB%AC%B4%EC%97%87%EC%9D%84+%EB%91%98+%EA%B2%83%EC%9D%B8%EA%B0%80%5D_%5B%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%5D_%5B%5D_%5B%EC%9D%91%EC%9A%A9+%EC%84%9C%EB%B9%84%EC%8A%A4%EC%99%80+%EB%8F%84%EB%A9%94%EC%9D%B8+%EC%84%9C%EB%B9%84%EC%8A%A4%EC%9D%98+%EC%97%AD%ED%95%A0%2C+%EA%B7%B8%EB%A6%AC%EA%B3%A0+DIP%EB%A5%BC+%EC%A0%81%EC%9A%A9%ED%95%9C+%EC%84%B8+%EA%B0%80%EC%A7%80+%EA%B5%AC%ED%98%84+%EB%B0%A9%EC%8B%9D+%EB%B9%84%EA%B5%90%5D_%5B%5D.md)에서 응용 서비스와 도메인 서비스를 설계하는 세 가지 방식을 비교했다. 방식 1(JPA 직접 DI), 방식 2(DIP 적용), 방식 3(파라미터 전달). 이론적으로 방식 2를 권장했지만, "그래서 실제로 뭐가 다른데?"라는 질문에는 답하지 못했다.

이번 글에서는 **같은 비즈니스 로직을 세 가지 방식으로 구현한 뒤, 각각에 대해 단위 테스트를 작성**해본다. 상태 검증, 행위 검증, 예외 검증, 경계값 테스트, 파라미터화 테스트까지. 테스트 코드가 얼마나 깔끔한지, 얼마나 깨지기 쉬운지를 직접 비교하면 어떤 설계가 좋은 설계인지 **테스트가 증명**해준다.

> 테스트가 어려우면 설계를 의심해라.

## 테스트 대상: 커머스 도메인

비교를 위해 동일한 비즈니스 시나리오를 사용한다. 커머스 서비스의 **재고 차감** 로직이다.

### 도메인 규칙

- 상품은 이름, 가격, 재고수량을 가진다.
- 재고 차감 시, 요청 수량이 현재 재고보다 크면 예외가 발생한다.
- 재고는 0 미만이 될 수 없다.
- 존재하지 않는 상품에 대한 차감 요청은 예외가 발생한다.

```kotlin
// domain/product/Product.kt
@Entity
class Product(
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long = 0L,
    var name: String,
    @Embedded var price: Money,
    @Embedded var stockQuantity: StockQuantity,
    val brandId: Long,
) {
    fun deductStock(quantity: Quantity) {
        stockQuantity = stockQuantity.deduct(quantity)
    }

    companion object {
        fun create(
            name: String,
            price: Money,
            stockQuantity: StockQuantity,
            brandId: Long,
        ): Product = Product(
            name = name,
            price = price,
            stockQuantity = stockQuantity,
            brandId = brandId,
        )
    }
}
```

```kotlin
// domain/product/StockQuantity.kt
@Embeddable
data class StockQuantity(val value: Int) {
    init {
        require(value >= 0) { "재고 수량은 0 이상이어야 합니다." }
    }

    fun deduct(quantity: Quantity): StockQuantity {
        if (value < quantity.value) {
            throw CoreException(ErrorType.BAD_REQUEST, "재고가 부족합니다. 현재: $value, 요청: ${quantity.value}")
        }
        return StockQuantity(value - quantity.value)
    }
}
```

이제 이 도메인을 다루는 **도메인 서비스**를 세 가지 방식으로 구현하고, 각각 테스트해본다.

## 방식 1: JPA 직접 의존 — 테스트의 고통

### 도메인 서비스 코드

```kotlin
// domain/product/ProductService.kt
@Service
class ProductService(
    private val productJpaRepository: ProductJpaRepository,  // 인프라 직접 의존
) {
    fun getProduct(productId: Long): Product {
        return productJpaRepository.findByIdAndDeletedAtIsNull(productId)
            ?: throw CoreException(ErrorType.NOT_FOUND, "상품을 찾을 수 없습니다.")
    }

    fun deductStocks(productIds: List<Long>, requests: List<StockDeductionRequest>) {
        val products = productJpaRepository.findAllByIdInAndDeletedAtIsNull(productIds)
        val productMap = products.associateBy { it.id }

        for (request in requests) {
            val product = productMap[request.productId]
                ?: throw CoreException(ErrorType.NOT_FOUND, "상품을 찾을 수 없습니다: ${request.productId}")
            product.deductStock(request.quantity)
        }
    }
}
```

### 테스트 작성

#### 상태 검증 테스트

```kotlin
@Test
fun `재고 차감 후 수량이 감소한다`() {
    // JpaRepository를 Mock해야 한다
    val mockJpaRepo = mockk<ProductJpaRepository>()
    val product = Product.create(
        name = "테스트 상품",
        price = Money(10000),
        stockQuantity = StockQuantity(10),
        brandId = 1L,
    ).apply {
        // id 필드를 설정하기 위해 리플렉션이 필요하다
        val idField = Product::class.java.getDeclaredField("id")
        idField.isAccessible = true
        idField.set(this, 1L)
    }

    every { mockJpaRepo.findAllByIdInAndDeletedAtIsNull(listOf(1L)) } returns listOf(product)

    val service = ProductService(mockJpaRepo)
    service.deductStocks(
        listOf(1L),
        listOf(StockDeductionRequest(1L, Quantity(3))),
    )

    // 상태 검증
    assertThat(product.stockQuantity.value).isEqualTo(7)
}
```

벌써 문제가 보인다.

1. **`ProductJpaRepository`를 Mock해야 한다.** 도메인 서비스를 테스트하는데 인프라 기술의 Mock이 필요하다.
2. **`findAllByIdInAndDeletedAtIsNull`이라는 인프라 메서드명을 테스트 코드에서 알아야 한다.** 이 메서드명이 바뀌면 테스트도 깨진다.
3. **엔티티의 `id`를 설정하기 위해 리플렉션이 필요하다.** JPA 엔티티는 `@GeneratedValue`로 ID가 자동 생성되므로 테스트에서 직접 설정하기 까다롭다.

#### 행위 검증 테스트

```kotlin
@Test
fun `재고 차감 시 repository에서 상품을 조회한다`() {
    val mockJpaRepo = mockk<ProductJpaRepository>()
    val product = createProductWithId(1L, stockQuantity = StockQuantity(10))

    every { mockJpaRepo.findAllByIdInAndDeletedAtIsNull(listOf(1L)) } returns listOf(product)

    val service = ProductService(mockJpaRepo)
    service.deductStocks(
        listOf(1L),
        listOf(StockDeductionRequest(1L, Quantity(3))),
    )

    // 행위 검증 — 어떤 메서드가 호출되었는지 확인
    verify { mockJpaRepo.findAllByIdInAndDeletedAtIsNull(listOf(1L)) }
}
```

`verify`로 `findAllByIdInAndDeletedAtIsNull`이 호출되었는지 검증하고 있다. 만약 JPA 쿼리 메서드 명명 규칙을 바꾸거나, `@Query` 어노테이션으로 전환하면? **비즈니스 로직은 안 바뀌었는데 테스트가 깨진다.**

#### 예외 검증 테스트

```kotlin
@Test
fun `존재하지 않는 상품을 조회하면 예외가 발생한다`() {
    val mockJpaRepo = mockk<ProductJpaRepository>()
    every { mockJpaRepo.findByIdAndDeletedAtIsNull(999L) } returns null  // 인프라 메서드명 노출

    val service = ProductService(mockJpaRepo)

    assertThatThrownBy { service.getProduct(999L) }
        .isInstanceOf(CoreException::class.java)
        .hasFieldOrPropertyWithValue("errorType", ErrorType.NOT_FOUND)
}
```

#### 경계값 테스트

```kotlin
@Nested
@DisplayName("재고 차감 경계값 테스트")
inner class StockDeductionBoundary {

    @Test
    fun `재고와 동일한 수량을 차감하면 성공한다 — 재고 0`() {
        val mockJpaRepo = mockk<ProductJpaRepository>()
        val product = createProductWithId(1L, stockQuantity = StockQuantity(5))
        every { mockJpaRepo.findAllByIdInAndDeletedAtIsNull(listOf(1L)) } returns listOf(product)

        val service = ProductService(mockJpaRepo)
        service.deductStocks(
            listOf(1L),
            listOf(StockDeductionRequest(1L, Quantity(5))),
        )

        assertThat(product.stockQuantity.value).isEqualTo(0)
    }

    @Test
    fun `재고보다 1개 더 차감하면 예외가 발생한다`() {
        val mockJpaRepo = mockk<ProductJpaRepository>()
        val product = createProductWithId(1L, stockQuantity = StockQuantity(5))
        every { mockJpaRepo.findAllByIdInAndDeletedAtIsNull(listOf(1L)) } returns listOf(product)

        val service = ProductService(mockJpaRepo)

        assertThatThrownBy {
            service.deductStocks(
                listOf(1L),
                listOf(StockDeductionRequest(1L, Quantity(6))),
            )
        }.isInstanceOf(CoreException::class.java)
    }
}
```

경계값 테스트인데 **매번 Mock 설정을 반복**하고 있다. `every { mockJpaRepo.findAllByIdInAndDeletedAtIsNull(...) }` 코드가 모든 테스트에 등장한다. 테스트의 핵심(경계값 검증)보다 **Mock 준비 코드가 더 많다.**

### 방식 1 테스트의 문제 요약

| 문제 | 설명 |
|------|------|
| Mock 보일러플레이트 | 모든 테스트에서 `mockk<ProductJpaRepository>()` 반복 |
| 인프라 메서드명 노출 | `findAllByIdInAndDeletedAtIsNull` 같은 세부사항이 테스트에 침투 |
| 리팩토링 내성 없음 | JPA 메서드 시그니처 변경 시 모든 테스트 깨짐 |
| 테스트 의도 불명확 | Mock 설정 코드가 많아서 "무엇을 테스트하는지" 파악이 어려움 |

## 방식 2: DIP 적용 — 테스트가 쉬워지는 설계

### 도메인 서비스 코드

```kotlin
// domain/product/ProductRepository.kt — 도메인 인터페이스
interface ProductRepository {
    fun save(product: Product): Product
    fun findById(id: Long): Product?
    fun findAllByIds(ids: List<Long>): List<Product>
    fun findAllByBrandId(brandId: Long): List<Product>
}
```

```kotlin
// domain/product/ProductService.kt
@Component
class ProductService(
    private val productRepository: ProductRepository,  // 도메인 인터페이스에만 의존
) {
    fun getProduct(productId: Long): Product {
        return productRepository.findById(productId)
            ?: throw CoreException(ErrorType.NOT_FOUND, "상품을 찾을 수 없습니다.")
    }

    fun deductStocks(productIds: List<Long>, requests: List<StockDeductionRequest>) {
        val products = productRepository.findAllByIds(productIds)
        val productMap = products.associateBy { it.id }

        for (request in requests) {
            val product = productMap[request.productId]
                ?: throw CoreException(ErrorType.NOT_FOUND, "상품을 찾을 수 없습니다: ${request.productId}")
            product.deductStock(request.quantity)
        }
    }
}
```

### Fake Repository 작성

DIP를 적용했으니 **Fake 구현체**를 만들 수 있다. 한 번 만들어두면 모든 테스트에서 재사용한다.

```kotlin
// test/fake/FakeProductRepository.kt
class FakeProductRepository : ProductRepository {
    private val store = mutableMapOf<Long, Product>()
    private var sequence = 1L

    override fun save(product: Product): Product {
        val id = if (product.id == 0L) sequence++ else product.id
        val saved = Product.create(
            name = product.name,
            price = product.price,
            stockQuantity = product.stockQuantity,
            brandId = product.brandId,
        ).also {
            val idField = Product::class.java.getDeclaredField("id")
            idField.isAccessible = true
            idField.set(it, id)
        }
        store[id] = saved
        return saved
    }

    override fun findById(id: Long): Product? = store[id]

    override fun findAllByIds(ids: List<Long>): List<Product> =
        ids.mapNotNull { store[it] }

    override fun findAllByBrandId(brandId: Long): List<Product> =
        store.values.filter { it.brandId == brandId }
}
```

Fake는 DB 없이 실제처럼 동작하는 인메모리 구현체다. Mock과 달리 **`when().thenReturn()` 설정이 필요 없다.**

### 테스트 작성

#### 상태 검증 테스트

```kotlin
@Test
fun `재고 차감 후 수량이 감소한다`() {
    val repository = FakeProductRepository()
    val product = repository.save(
        Product.create(
            name = "테스트 상품",
            price = Money(10000),
            stockQuantity = StockQuantity(10),
            brandId = 1L,
        ),
    )

    val service = ProductService(repository)
    service.deductStocks(
        listOf(product.id),
        listOf(StockDeductionRequest(product.id, Quantity(3))),
    )

    // 상태 검증 — 깔끔하다
    val found = repository.findById(product.id)!!
    assertThat(found.stockQuantity.value).isEqualTo(7)
}
```

방식 1과 비교해보자.

- `mockk<>()`, `every { ... }` 같은 Mock 설정이 없다.
- `findAllByIdInAndDeletedAtIsNull` 같은 인프라 메서드명이 없다.
- **테스트가 "무엇을 검증하는지" 한눈에 보인다.** 상품을 저장하고, 재고를 차감하고, 결과를 확인한다.

#### 행위 검증 테스트 — 꼭 필요한 경우에만

```kotlin
@Test
fun `재고 차감 시 repository에서 상품을 조회한다`() {
    // Mock을 써야 하지만, 도메인 인터페이스의 Mock이다
    val mockRepository = mockk<ProductRepository>()
    val product = createProduct(id = 1L, stockQuantity = StockQuantity(10))

    every { mockRepository.findAllByIds(listOf(1L)) } returns listOf(product)

    val service = ProductService(mockRepository)
    service.deductStocks(
        listOf(1L),
        listOf(StockDeductionRequest(1L, Quantity(3))),
    )

    // 행위 검증 — 도메인 언어로 표현된다
    verify { mockRepository.findAllByIds(listOf(1L)) }
}
```

행위 검증이 필요할 때도 `findAllByIds`라는 **도메인 언어**로 검증한다. `findAllByIdInAndDeletedAtIsNull`이 아니다. JPA 메서드 시그니처가 바뀌어도 이 테스트는 깨지지 않는다.

하지만 기억하자. **행위 검증은 "반드시 호출되어야 하는 비즈니스 요구사항"이 있을 때만 사용한다.** 위 테스트는 사실 상태 검증으로 충분하다. 재고가 줄었으면 조회가 되었다는 뜻이니까.

#### 예외 검증 테스트

```kotlin
@Test
fun `존재하지 않는 상품을 조회하면 예외가 발생한다`() {
    val repository = FakeProductRepository()
    val service = ProductService(repository)

    // Fake를 쓰면 "비어있는 저장소에서 조회" — 자연스럽다
    assertThatThrownBy { service.getProduct(999L) }
        .isInstanceOf(CoreException::class.java)
        .hasFieldOrPropertyWithValue("errorType", ErrorType.NOT_FOUND)
}
```

Mock으로 `returns null`을 설정하는 것보다 **"비어있는 저장소에서 조회하면 없다"**가 더 자연스럽고 읽기 쉽다.

```kotlin
@Test
fun `재고 차감 시 일부 상품이 존재하지 않으면 예외가 발생한다`() {
    val repository = FakeProductRepository()
    val product = repository.save(
        Product.create(name = "상품A", price = Money(10000), stockQuantity = StockQuantity(10), brandId = 1L),
    )

    val service = ProductService(repository)

    // 존재하는 상품(product.id)과 존재하지 않는 상품(999L)을 함께 요청
    assertThatThrownBy {
        service.deductStocks(
            listOf(product.id, 999L),
            listOf(
                StockDeductionRequest(product.id, Quantity(1)),
                StockDeductionRequest(999L, Quantity(1)),
            ),
        )
    }.isInstanceOf(CoreException::class.java)
        .hasMessageContaining("999")
}
```

#### 경계값 테스트

```kotlin
@Nested
@DisplayName("재고 차감 경계값 테스트")
inner class StockDeductionBoundary {
    private val repository = FakeProductRepository()
    private val service = ProductService(repository)

    @Test
    fun `재고와 동일한 수량을 차감하면 성공한다 — 재고 0`() {
        val product = repository.save(
            Product.create(name = "상품", price = Money(10000), stockQuantity = StockQuantity(5), brandId = 1L),
        )

        service.deductStocks(
            listOf(product.id),
            listOf(StockDeductionRequest(product.id, Quantity(5))),
        )

        assertThat(repository.findById(product.id)!!.stockQuantity.value).isEqualTo(0)
    }

    @Test
    fun `재고보다 1개 더 차감하면 예외가 발생한다`() {
        val product = repository.save(
            Product.create(name = "상품", price = Money(10000), stockQuantity = StockQuantity(5), brandId = 1L),
        )

        assertThatThrownBy {
            service.deductStocks(
                listOf(product.id),
                listOf(StockDeductionRequest(product.id, Quantity(6))),
            )
        }.isInstanceOf(CoreException::class.java)
    }

    @Test
    fun `재고가 0인 상품에서 1개를 차감하면 예외가 발생한다`() {
        val product = repository.save(
            Product.create(name = "상품", price = Money(10000), stockQuantity = StockQuantity(0), brandId = 1L),
        )

        assertThatThrownBy {
            service.deductStocks(
                listOf(product.id),
                listOf(StockDeductionRequest(product.id, Quantity(1))),
            )
        }.isInstanceOf(CoreException::class.java)
    }
}
```

Mock 설정이 없으니 **경계값에만 집중**할 수 있다. `repository`와 `service`를 필드로 선언해두면 반복 코드도 줄어든다.

#### 파라미터화 테스트

```kotlin
@ParameterizedTest
@CsvSource(
    "10, 1, 9",
    "10, 5, 5",
    "10, 10, 0",
    "100, 99, 1",
    "1, 1, 0",
)
@DisplayName("다양한 재고 차감 시나리오")
fun `재고 차감 후 남은 수량이 정확하다`(
    initialStock: Int,
    deductQuantity: Int,
    expectedRemaining: Int,
) {
    val repository = FakeProductRepository()
    val product = repository.save(
        Product.create(
            name = "상품",
            price = Money(10000),
            stockQuantity = StockQuantity(initialStock),
            brandId = 1L,

    val service = ProductService(repository)

    assertThat(repository.findById(product.id)!!.stockQuantity.value).isEqualTo(expectedRemaining)
}
```

```kotlin

    "0, 1",
```

파라미터화 테스트에서 Fake의 장점이 극대화된다. Mock이었다면 **매 파라미터마다 `every { ... }` 설정을 다시 해야** 한다. Fake는 그냥 저장하고 차감하면 된다.

## 방식 3: 파라미터 전달 — 순수 함수의 아름다움과 그 대가

### 도메인 서비스 코드

```kotlin
// domain/product/ProductService.kt
@Component
class ProductService {
    // Repository 의존성이 없다. 완전히 순수하다.

    fun deductStocks(products: Map<Long, Product>, requests: List<StockDeductionRequest>) {
        for (request in requests) {
            val product = products[request.productId]
                ?: throw CoreException(ErrorType.NOT_FOUND, "상품을 찾을 수 없습니다: ${request.productId}")
            product.deductStock(request.quantity)
        }
    }
}
```

### 테스트 작성

#### 상태 검증 테스트

```kotlin
@Test
fun `재고 차감 후 수량이 감소한다`() {
    // 의존성 없이 바로 생성
    val service = ProductService()
    val product = Product.create(
        name = "테스트 상품",
        price = Money(10000),
        stockQuantity = StockQuantity(10),
        brandId = 1L,
    )

    service.deductStocks(
        mapOf(1L to product),
        listOf(StockDeductionRequest(1L, Quantity(3))),
    )

    // 상태 검증 — 가장 깔끔하다
    assertThat(product.stockQuantity.value).isEqualTo(7)
}
```

이건 진짜 깔끔하다. Mock도 Fake도 없다. `ProductService()`에 생성자 인자가 없다. **순수 객체를 만들고, 메서드를 호출하고, 결과를 확인한다.** 이보다 더 단순한 테스트는 없다.

#### 예외 검증 테스트

```kotlin
@Test
fun `존재하지 않는 상품에 대한 차감 요청은 예외가 발생한다`() {
    val service = ProductService()
    val product = Product.create(
        name = "상품A",
        price = Money(10000),
        stockQuantity = StockQuantity(10),

    assertThatThrownBy {
        service.deductStocks(
            mapOf(1L to product),

}
```

#### 경계값 테스트


**방식 3의 테스트가 가장 깔끔하다.** 이건 부정할 수 없다.

### 그런데 — 응용 서비스의 테스트는?

방식 3은 도메인 서비스 테스트가 아름답지만, **응용 서비스가 그 고통을 떠안는다.**

```kotlin
// application/product/ProductFacade.kt
@Component
class ProductFacade(
    private val productJpaRepository: ProductJpaRepository,  // 인프라 직접 의존
    private val productService: ProductService,
) {
    @Transactional
    fun deductStocks(requests: List<StockDeductionRequest>) {
        val productIds = requests.map { it.productId }
        val products = productJpaRepository.findAllByIdInAndDeletedAtIsNull(productIds)
        val productMap = products.associateBy { it.id }

        productService.deductStocks(productMap, requests)
    }
}
```

```kotlin
// 응용 서비스 테스트 — 결국 JPA Mock이 필요하다
@Test
fun `재고 차감 유스케이스`() {
    val mockJpaRepo = mockk<ProductJpaRepository>()
    val productService = ProductService()
    val product = createProductWithId(1L, stockQuantity = StockQuantity(10))

    // 다시 인프라 메서드명이 등장한다
    every { mockJpaRepo.findAllByIdInAndDeletedAtIsNull(listOf(1L)) } returns listOf(product)

    val facade = ProductFacade(mockJpaRepo, productService)
    facade.deductStocks(listOf(StockDeductionRequest(1L, Quantity(3))))

    assertThat(product.stockQuantity.value).isEqualTo(7)
}
```

도메인 서비스에서 몰아낸 인프라 의존성이 **응용 서비스 테스트에 그대로 나타난다.** 문제가 사라진 게 아니라 이동한 것뿐이다.

### 방식 3 + DIP 적용

이 문제를 개선할 수 있다. 응용 서비스가 도메인 인터페이스에 의존하게 바꾸면 된다.

```kotlin
@Component
class ProductFacade(
    private val productRepository: ProductRepository,  // 도메인 인터페이스
    private val productService: ProductService,
) {
    @Transactional
    fun deductStocks(requests: List<StockDeductionRequest>) {
        val productIds = requests.map { it.productId }
        val products = productRepository.findAllByIds(productIds)
        val productMap = products.associateBy { it.id }

        productService.deductStocks(productMap, requests)
    }
}
```

```kotlin
// 응용 서비스 테스트 — Fake 활용
@Test
fun `재고 차감 유스케이스`() {
    val repository = FakeProductRepository()
    val product = repository.save(
        Product.create(name = "상품", price = Money(10000), stockQuantity = StockQuantity(10), brandId = 1L),
    )
    val productService = ProductService()

    val facade = ProductFacade(repository, productService)
    facade.deductStocks(listOf(StockDeductionRequest(product.id, Quantity(3))))

    assertThat(repository.findById(product.id)!!.stockQuantity.value).isEqualTo(7)
}
```

인프라 메서드명이 사라지고 깔끔해졌다. 하지만 **도메인 서비스와 응용 서비스의 역할 경계**가 모호해지는 문제는 남는다.

## 테스트 종류별 비교: 같은 시나리오, 다른 코드

같은 비즈니스 규칙 — "재고가 부족하면 차감할 수 없다" — 을 세 방식으로 테스트했을 때, 코드가 어떻게 다른지 나란히 놓고 보자.

### 상태 검증 테스트 비교

```kotlin
// 방식 1: JPA 직접 의존
val mockJpaRepo = mockk<ProductJpaRepository>()
every { mockJpaRepo.findAllByIdInAndDeletedAtIsNull(listOf(1L)) } returns listOf(product)
val service = ProductService(mockJpaRepo)
service.deductStocks(listOf(1L), listOf(StockDeductionRequest(1L, Quantity(3))))
assertThat(product.stockQuantity.value).isEqualTo(7)

// 방식 2: DIP 적용
val repository = FakeProductRepository()
val product = repository.save(Product.create(...))
val service = ProductService(repository)
service.deductStocks(listOf(product.id), listOf(StockDeductionRequest(product.id, Quantity(3))))
assertThat(repository.findById(product.id)!!.stockQuantity.value).isEqualTo(7)

// 방식 3: 파라미터 전달
val service = ProductService()
val product = Product.create(...)
service.deductStocks(mapOf(1L to product), listOf(StockDeductionRequest(1L, Quantity(3))))
assertThat(product.stockQuantity.value).isEqualTo(7)
```

### 핵심 차이

**방식 1**은 테스트 준비 단계에서 `every { mockJpaRepo.findAllByIdInAndDeletedAtIsNull(...) }`라는 **인프라의 세부사항을 알아야** 한다. 이 메서드명이 바뀌면 테스트가 깨진다.

**방식 2**는 Fake를 통해 "저장하고 조회하는" 자연스러운 흐름으로 테스트한다. 인프라 메서드명이 테스트에 등장하지 않는다.

**방식 3**은 가장 단순하지만, 테스트 대상이 "데이터가 이미 준비된 상태에서의 로직"만 검증한다. 데이터 조회 ~ 로직 수행까지의 **전체 흐름을 검증하지 못한다.**

## 리팩토링 내성 비교: 진짜 중요한 기준

> 좋은 테스트의 가장 중요한 속성은 **리팩토링 내성**이다. 구현을 바꿨는데 비즈니스 로직은 안 변했다면, 테스트도 통과해야 한다.

JPA의 `findAllByIdInAndDeletedAtIsNull`을 QueryDSL로 바꾸는 상황을 가정해보자.

```kotlin
// Before: Spring Data JPA
fun findAllByIdInAndDeletedAtIsNull(ids: List<Long>): List<Product>

// After: QueryDSL
fun findActiveProductsByIds(ids: List<Long>): List<Product>
```

| | 방식 1 | 방식 2 | 방식 3 |
|---|---|---|---|
| **도메인 서비스 코드 변경** | O (메서드명 변경) | X (인터페이스 불변) | X (의존성 없음) |
| **도메인 서비스 테스트 깨짐** | O (Mock 메서드명 변경) | X | X |
| **응용 서비스 코드 변경** | — | X | O (메서드명 변경) |
| **응용 서비스 테스트 깨짐** | — | X | O |
| **인프라 구현체 변경** | — | O (의도된 변경) | — |

**방식 2만 유일하게 "비즈니스 로직 계층의 코드와 테스트가 전혀 안 깨진다."** 변경이 인프라 구현체에 국한된다. 이것이 DIP의 실질적인 가치다.

## 종합 비교

| 비교 항목 | 방식 1: JPA 직접 DI | 방식 2: DIP 적용 | 방식 3: 파라미터 전달 |
|----------|-------------------|-----------------|-------------------|
| **상태 검증 테스트** | Mock 설정 필요 | Fake로 자연스러움 | 가장 단순 |
| **행위 검증 테스트** | 인프라 메서드명 노출 | 도메인 언어로 검증 | 해당 없음 (의존성 없음) |
| **예외 검증 테스트** | `returns null` 설정 | 빈 저장소 활용 | 빈 Map 전달 |
| **경계값 테스트** | 매번 Mock 반복 | Fake 재사용 | 가장 깔끔 |
| **파라미터화 테스트** | Mock 설정이 테스트마다 필요 | Fake로 깔끔 | 가장 깔끔 |
| **리팩토링 내성** | 낮음 | 높음 | 높음 (도메인만) |
| **Mock 보일러플레이트** | 많음 | 적음 (Fake 사용 시 없음) | 없음 |
| **전체 흐름 검증** | 가능 | 가능 | 불가 (응용서비스에서 해야 함) |

## 마무리: 어떤 설계가 좋은 테스트를 만드는가

세 가지 방식을 모두 테스트해본 결과, 답은 명확하다.

### 방식 1은 피해야 한다.

Mock 설정의 반복, 인프라 메서드명의 침투, 리팩토링 시 깨지는 테스트. 세 가지 문제가 동시에 발생한다. 테스트를 작성할수록 **유지보수 부채가 쌓인다.**

### 방식 3은 매력적이지만, 절반의 답이다.

도메인 서비스 테스트는 아름답다. 하지만 응용 서비스가 그 고통을 대신 치른다. 도메인 서비스만 놓고 보면 최고지만, **시스템 전체의 테스트 품질**은 방식 2가 더 낫다.

### 방식 2가 균형 잡힌 답이다.

- Fake 구현체 하나로 Mock 보일러플레이트를 제거한다.
- 도메인 인터페이스 덕분에 인프라 세부사항이 테스트에 침투하지 않는다.
- 리팩토링 내성이 높다. 기술을 교체해도 비즈니스 로직 테스트가 안 깨진다.
- 도메인 서비스가 조회부터 로직 수행까지 **자기 완결적으로 동작**하므로, 하나의 테스트로 전체 흐름을 검증할 수 있다.

> 인터페이스 하나 더 만드는 비용이 아깝다고 느낄 수 있다. 하지만 Mock 설정을 매번 반복하고, 메서드명이 바뀔 때마다 테스트를 고치는 비용과 비교해보라. **Fake Repository 하나가 수십 개의 테스트를 깔끔하게 만든다.**

### 최종 권장 조합

```
도메인 서비스 테스트 → Fake Repository + 상태 검증 (기본)
                     Mock + 행위 검증 (반드시 호출되어야 하는 비즈니스 규칙만)
                     경계값 + 파라미터화 테스트 (도메인 규칙 경계 검증)

응용 서비스 테스트  → Fake Repository + Mock 도메인 서비스 조합
                     유스케이스 흐름 검증에 집중
```

**좋은 설계는 좋은 테스트를 만든다.** 그리고 좋은 테스트는 설계가 올바른지 증명해준다. 테스트를 작성하면서 "이거 왜 이렇게 복잡하지?"라는 생각이 든다면, 테스트가 아니라 설계를 의심해야 한다.

# WIL - 동시성 문제를 이론에서 실전까지 관통한 한 주

## 이번 주 요약

이번 주는 **동시성**이라는 하나의 주제가 이론 학습과 프로젝트 구현 양쪽에서 동시에 터진 한 주였다. 글로 정리하면서 개념을 잡고, 커머스 프로젝트에 직접 적용하면서 체감하는 순환이 만들어졌다.

## 이번 주에 정리한 것

### 테스트가 증명하는 좋은 도메인 서비스 설계

도메인 서비스를 설계하는 세 가지 방식(JPA 직접 DI, DIP 적용, 파라미터 전달)을 **동일한 비즈니스 로직에 대해 단위 테스트를 작성**하며 비교했다.

- **방식 1(JPA 직접 DI)**: Mock 보일러플레이트가 매 테스트마다 반복되고, `findAllByIdInAndDeletedAtIsNull` 같은 인프라 메서드명이 테스트에 침투한다. 리팩토링하면 테스트가 깨진다.
- **방식 2(DIP 적용)**: Fake Repository 하나로 Mock 설정이 사라진다. 인프라 세부사항이 테스트에 노출되지 않아 리팩토링 내성이 높다.
- **방식 3(파라미터 전달)**: 도메인 서비스 테스트는 가장 깔끔하지만, 응용 서비스가 그 고통을 대신 치른다.

결론은 **방식 2가 시스템 전체의 테스트 품질에서 가장 균형 잡힌 답**이라는 것이다. "테스트가 어려우면 설계를 의심하라"는 말을 코드로 증명한 글이었다.

---

### 동시성과 Lock의 개념 - 순수 Java편

동시성 문제의 근본 원인 세 가지(경쟁 조건, 가시성, 명령어 재배치)를 정리하고, Java가 제공하는 동시성 제어 도구를 다뤘다.

- `synchronized`: 상호 배제 + 가시성 + 재배치 방지를 한 번에 보장하지만, 유연성 부족
- `volatile`: 가시성만 보장. 복합 연산에는 무력
- `ReentrantLock`: tryLock, 공정성, 인터럽트 등 세밀한 제어
- `ReadWriteLock`: 읽기-쓰기 분리로 읽기 위주 워크로드에서 처리량 향상
- **CAS(Compare-And-Swap)**: CPU 수준의 원자적 연산. 경쟁이 낮을 때 Lock-Free로 높은 처리량

핵심은 **경쟁 강도에 따라 도구를 선택**하는 것이다. CAS는 경쟁이 낮을 때, Lock은 경쟁이 심할 때 유리하다.

---

### 동시성과 Lock의 개념 - DB와 JPA편

애플리케이션 레벨에서 DB/JPA 레벨로 확장했다.

- **DB 격리 수준**: Dirty Read, Non-Repeatable Read, Phantom Read 세 가지 읽기 이상 현상과 격리 수준별 방지 범위
- **낙관적 락**: `@Version`으로 커밋 시 충돌 감지. CAS와 같은 원리
- **비관적 락**: `SELECT FOR UPDATE`로 즉시 잠금. `synchronized`와 같은 원리

Java의 `synchronized` ↔ 비관적 락, CAS ↔ 낙관적 락, `volatile` ↔ 1차 캐시 clear라는 **레벨만 다를 뿐 같은 질문**이라는 대응 관계가 인상적이었다.

## 이번 주에 구현한 것

커머스 프로젝트에서 동시성 관련 이슈를 집중적으로 해결했다.

| 문제 | 해결 방식 |
|------|----------|
| 좋아요 카운트 Lost Update | 원자적 UPDATE 쿼리 적용 |
| 좋아요 / 쿠폰 중복 등록 (TOCTOU) | UNIQUE 제약 + 예외 변환 |
| 재고 차감 동시성 문제 | DB 비관적 락 (`SELECT FOR UPDATE`) |
| 주문 상태 변경 Lost Update | `@Version` 낙관적 락 |
| 중복 주문 방지 | Idempotency-Key 헤더 기반 멱등성 구현 |
| UK 기반 동시성 제어의 한계 | UK 제거 + 비관적 락으로 전환 |

동시성 테스트도 작성했다. 10개 스레드가 동시에 같은 상품에 주문을 넣어도 재고가 정확히 차감되는지, 재고 초과 시 음수가 되지 않는지를 검증했다.

# WIL - 외부 장애에서 살아남는 결제 시스템을 설계한 한 주

## 이번 주 요약

이번 주는 **외부 시스템 연동 결제를 처음부터 끝까지 설계하고 구현**한 한 주였다. 결제 도메인 모델을 만들고, 외부 시스템과 연동하고, 실패를 예외가 아닌 값으로 다루는 Failure-Ready 패턴을 적용했다. CircuitBreaker, Bulkhead, Retry로 외부 장애 전파를 차단하고, 콜백 보상 트랜잭션으로 주문-결제 정합성을 맞췄다. 마지막에는 HMAC-SHA256 서명 검증 필터로 보안까지 챙겼다. "외부 시스템은 언제든 실패한다"는 전제 위에 세운 결제 시스템을, 글로 정리하고 코드로 증명한 한 주였다.

---

## 이번 주에 정리한 것

### 일단 지키고 보자 — 외부 시스템 장애에 대처하는 코드의 태도

외부 시스템 서버가 죽으면 우리 결제 시스템도 죽어야 하는 걸까? 이 질문에서 시작한 글이었다. 기술보다 먼저 정해야 하는 건 **태도**라는 결론을 내렸다.

- **실패를 값으로 표현한다**: `PaymentGateway`의 반환 타입을 nullable로 선언. 예외가 아니라 null이 오면, try-catch가 아니라 null 체크로 분기한다. 실패가 프로그램의 중단이 아니라 분기 조건이 된다.
- **실패해도 상태를 기록한다**: 외부 시스템 호출 전에 REQUESTED로 저장하고, 성공이든 실패든 최종 상태를 DB에 반영한다. 어떤 시나리오에서도 결제 기록은 남는다.
- **남의 장애가 나한테 번지지 않게 막는다**: Bulkhead로 외부 시스템 호출 스레드를 제한하고, CircuitBreaker로 반복 실패 시 호출을 중단한다.
- **재시도에도 기준이 있다**: 멱등한 조회만 재시도하고, 비멱등한 결제 요청은 재시도하지 않는다. "해도 되는가?"를 먼저 묻고 기술을 적용한다.

결국 CircuitBreaker든 Retry든 Bulkhead든, 그 밑에 깔려 있는 건 하나의 원칙이다 — **"일단 나부터 살자."**

## 이번 주에 구현한 것

이번 주 커밋: 20건 / 50개 파일 변경 / +4,315줄

### 1. 외부 시스템 연동 결제 전체 구현

결제 도메인을 처음부터 설계했다.

| 구성 요소 | 설명 |
|----------|------|
| `Payment` 엔티티 | 상태 머신 기반 결제 도메인 모델. REQUESTED → PENDING → SUCCESS/FAILED |
| `PaymentGateway` 인터페이스 | 도메인 레이어에 위치. nullable 반환으로 "응답이 없을 수 있다"를 타입에 녹임 |
| `PaymentGatewayImpl` | Feign + Fallback으로 외부 시스템 통신. 모든 예외를 null로 변환 |
| `PaymentFacade` | 결제 요청 → 외부 시스템 호출 → 실패 시 재조회 → 상태 기록의 전체 오케스트레이션 |

핵심은 `PaymentFacade`의 흐름이다. 외부 시스템이 응답하지 않으면 한 번 더 재조회하고, 그래도 안 되면 FAILED로 기록한다. **어떤 경우에도 결제 기록은 DB에 남고, 시스템은 살아있다.**

### 2. Failure-Ready 패턴 + Resilience4j

외부 장애에 대비하는 세 가지 장치를 적용했다.

| 패턴 | 적용 | 역할 |
|------|------|------|
| **Bulkhead** | `pg-simulator` | 외부 시스템 동시 요청을 25개로 제한. 나머지 스레드는 다른 API에 사용 |
| **CircuitBreaker** | `pg-simulator` | 최근 10건 중 50% 이상 실패 시 10초간 호출 차단 |
| **Retry** | `pg-query` (조회 전용) | 멱등한 조회만 최대 3회 재시도. 지수 백오프 적용 |

CircuitBreaker 그룹 공유 설정도 추가해서, requestPayment와 getTransactionStatus가 같은 CircuitBreaker 인스턴스를 공유하도록 했다.

### 3. 주문-결제 연동 및 콜백 보상 트랜잭션

결제 콜백이 FAILED로 들어오면, 주문을 취소하면서 **재고와 쿠폰을 복구하는 보상 트랜잭션**을 구현했다.

- 콜백 데이터를 그대로 믿지 않고, 외부 시스템에 직접 재조회해서 검증
- 외부 시스템 재조회 실패 시 검증되지 않은 콜백으로 상태 변경하는 버그도 수정

### 4. 외부 시스템 콜백 HMAC-SHA256 서명 검증 필터

콜백 API는 외부에서 호출하므로 위변조 방지가 필수다. `PgCallbackSignatureFilter`를 구현해서, 외부 시스템이 보낸 HMAC-SHA256 서명을 검증하도록 했다.

- 공유 시크릿 키 기반 서명 검증
- 필터 에러 응답 로직을 `FilterErrorResponseWriter` 확장 함수로 추출해서 기존 인증 필터와 공통화

### 5. 테스트 전방위 작성

이번 주에 가장 많은 시간을 쓴 부분이다.

| 테스트 종류 | 내용 |
|------------|------|
| **단위 테스트** | Payment 상태 전이, PaymentFacade 분기 로직 |
| **통합 테스트** | PaymentService DB 연동, PaymentFacade 전체 흐름 |
| **E2E 테스트** | 결제 요청 → 콜백 → 동기화 전 과정 API 레벨 검증 |
| **동시성 테스트** | CircuitBreaker 동시 요청 시 상태 전이 정합성 |
| **부하 테스트** | k6 스크립트로 CircuitBreaker 동작 시나리오별 부하 테스트 |
| **정책 테스트** | Resilience4j 설정이 의도대로 적용됐는지 검증 |

E2E 테스트에서 하드코딩된 orderId로 인한 false-positive도 발견해서 수정했다. **테스트가 통과하는 것보다 테스트가 진짜 검증하고 있는지가 더 중요하다.**

### 6. 모니터링

- Grafana 대시보드로 CircuitBreaker 상태, 호출 성공/실패율을 시각화
- k6 부하 테스트 결과를 문서로 정리

### 버그 수정 요약

| 문제 | 수정 |
|------|------|
| 외부 시스템 즉시 FAILED 응답 시 과거 거래가 재사용됨 | FAILED 응답을 별도 분기로 분리 |
| 외부 시스템 재조회 실패 시 미검증 콜백으로 상태 변경됨 | 재조회 실패 시 콜백 데이터도 차단 |
| E2E 테스트 orderId 하드코딩으로 false-positive | 테스트마다 고유 orderId 생성 |

## 이번 주의 깨달음

이번 주는 **"외부 시스템을 호출하는 코드에는 태도가 필요하다"**는 것을 체감한 한 주였다.

결제 시스템을 처음 설계할 때 가장 먼저 정한 건 기술이 아니라 원칙이었다. "외부 시스템이 응답하지 않을 수 있다"는 전제를 받아들이자, 코드의 흐름이 완전히 달라졌다. 예외를 던지는 대신 null을 반환하고, try-catch 대신 null 체크로 분기하고, 실패해도 기록을 남기는 코드가 자연스럽게 나왔다.

**태도가 먼저고, 기술은 그 다음이다.** CircuitBreaker와 Bulkhead는 "남의 장애가 나한테 번지면 안 된다"는 태도의 구현이고, Retry 정책은 "재시도해도 되는가?"라는 질문의 구현이고, HMAC 서명 검증은 "외부에서 오는 데이터는 믿지 않는다"는 태도의 구현이다.

그리고 테스트. 결제 시스템처럼 실패 시나리오가 복잡한 도메인에서는, **테스트가 곧 명세서**다. "외부 시스템이 타임아웃 나면 어떻게 되지?"라는 질문에 코드로 답하고, 그 답이 맞는지 테스트로 증명했다. 단위 테스트부터 E2E, 동시성, 부하 테스트까지 전방위로 작성하면서, 각 계층에서 어떤 시나리오를 검증해야 하는지 감을 잡았다.

결제 시스템을 만들면서 깨달은 건, 결국 **좋은 코드란 "잘 되는 경우"만 다루는 코드가 아니라 "안 되는 경우"까지 설계한 코드**라는 것이다.

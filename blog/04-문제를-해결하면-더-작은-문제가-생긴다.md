# 문제를 해결하면 더 작은 문제가 생긴다 — Thundering Herd가 가르쳐주는 것

## 핵심 질문

> 대기열로 10,000명의 동시 요청을 해결했더니, 175명의 동시 요청이라는 **축소판 문제**가 다시 등장한다. 이건 해결인가, 전이인가?

## 다루고 싶은 것

- "문제를 한 단계 뒤로 미룬 것"과 "문제를 해결한 것"의 차이
- 스케줄러가 1초마다 175명에게 토큰을 발급하면 175명이 동시에 주문 API를 호출하는 구조
- 캐시 만료 시 모든 요청이 동시에 DB를 조회하는 것과 같은 원리 — Thundering Herd
- 완화 전략들이 왜 **완전한 해결이 아니라 평탄화(smoothing)**인지를 정직하게 다루기
  - 발급 간격 분산: 1초 175명 → 100ms 18명
  - 토큰에 Jitter 부여: 활성화 시점 랜덤 딜레이
  - 주문 API 자체 Rate Limit: 최종 안전장치
- 대기열은 피크를 평탄화하는 것이지, 부하를 없애는 것이 아니라는 본질

## 출발점

- 대기열 설계 후 "문제 해결됐다"고 생각하는 순간의 함정
- 하류 시스템의 한계를 항상 염두에 두고 설계해야 하는 이유

# 과거는 현재보다 덜 중요해야 한다 — 랭킹에서 시간을 다루는 두 가지 설계

## 들어가며

커머스 서비스에서 "오늘의 인기 상품" 랭킹을 만든다고 하자. 유저 행동 이벤트(조회, 좋아요, 주문)를 수집하고, 점수를 매기고, 높은 순서대로 보여주면 된다. 단순하다.

그래서 Redis ZSET에 상품별 점수를 누적한다. 조회 +0.1, 좋아요 +0.2, 주문 +0.7. 시간이 지나면서 점수가 쌓이고, 인기 상품이 상위에 올라간다. 잘 동작하는 것처럼 보인다.

6개월이 지났다. 어느 날 신상품이 등록된다. 하루 만에 조회 500건, 주문 100건. 꽤 좋은 반응이다. 그런데 랭킹을 보면 이 상품은 어디에도 없다. 상위권에는 6개월 전부터 점수를 쌓아온 상품들이 자리를 잡고 있고, 하루치 점수로는 그 벽을 넘을 수 없다.

이건 불공정한 게 아니다. **관련 없는 것이 상위를 차지하고 있는 것이다.** "오늘의 인기 상품"이라고 해놓고, 실제로는 "역대 누적 상품"을 보여주고 있다. 유저가 원하는 건 공정한 랭킹이 아니라 **지금 자기에게 관련 있는 상품**이다.

이 글에서는 세 가지를 다룬다.

1. 누적이 왜 "관련성"을 망가뜨리는가
2. 과거의 영향력을 줄이는 두 가지 설계 — 자르기와 녹이기
3. 어떤 방식을 골라도 남는 문제는 무엇인가

---

## 누적의 함정 — 관련 없는 것이 상위를 차지한다

### 같은 판에서 경쟁할 수 없다

구체적인 숫자로 보자.

```
상품 A: 6개월 전 등록, 누적 조회 100,000건, 주문 5,000건
상품 B: 오늘 등록, 하루 조회 500건, 주문 100건

가중치: view=0.1, like=0.2, order=0.7

상품 A 점수: 100,000 × 0.1 + 5,000 × 0.7 = 13,500
상품 B 점수: 500 × 0.1 + 100 × 0.7 = 120
```

상품 B가 오늘 더 인기 있어도, 하루치 점수 120으로 6개월치 점수 13,500을 이기는 것은 구조적으로 불가능하다. 상품 B가 상품 A를 따라잡으려면 단순 계산으로 112일이 걸린다. 그 사이에 상품 A도 점수가 쌓이니 실제로는 영원히 따라잡을 수 없다.

상품 A가 나쁜 상품이라서 문제가 아니다. 상품 A가 **6개월 전에 인기 있었던 상품**이라서 문제다. "오늘의 인기"를 보여주겠다고 해놓고, 6개월 전의 인기를 보여주는 것은 **목적과 수단의 불일치**다.

### 노출이 노출을 만든다

더 심각한 문제는 **랭킹이 자기 강화적**이라는 점이다.

```
[상위 랭킹 상품]
     ↓ 노출 많음
     ↓ 클릭 많음
     ↓ 구매 많음
     ↓ 점수 더 높아짐
     ↓ 랭킹 더 올라감
     → 반복

[하위 랭킹 상품]
     ↓ 노출 없음
     ↓ 클릭 없음
     ↓ 구매 없음
     ↓ 점수 정체
     → 반복
```

랭킹이 높은 상품은 더 많이 노출되고, 더 많이 클릭되고, 더 많이 구매된다. 점수는 더 올라간다. 반대로, 랭킹에 진입하지 못한 상품은 노출 자체가 되지 않으니 클릭도 구매도 발생하지 않는다.

**점수가 높아서 노출되는 것이 아니라, 노출되어서 점수가 높아지는 구조.** 이것을 Matthew Effect(부익부 빈익빈)라고 부른다. 소수의 상품이 상위권을 독식하고, 나머지는 존재하지 않는 것과 같아진다.

이 문제는 나중에 다시 돌아온다. 기억해두자.

## 과거의 영향력을 줄이는 두 가지 길

누적의 문제를 한 문장으로 줄이면 이렇다.

> **과거가 현재보다 너무 강하다.**

그렇다면 과거의 영향력을 줄여야 한다. 방법은 크게 두 가지다.

| | 자르기 (시간의 양자화) | 녹이기 (시간 감쇠) |
|---|---|---|
| 핵심 아이디어 | 시간을 구간으로 나누고, 구간마다 0에서 시작 | 점수에 시간 함수를 곱해 과거일수록 줄어들게 |
| 과거 처리 방식 | 구간이 바뀌면 **끊는다** | 시간이 지나면 **줄인다** |
| 경계 | 명확 (자정에 리셋) | 없음 (연속적) |
| 비유 | 매일 새 시험지를 푸는 것 | 어제 시험 점수가 서서히 바래는 것 |

같은 문제에 대한 두 가지 접근이다. 둘 다 "과거는 현재보다 덜 중요해야 한다"는 같은 전제에서 출발한다. 차이는 과거를 **끊을 것인가, 줄일 것인가**에 있다.

## 시간을 자른다 — 양자화

### 매일 새로운 판이 열린다

```
ranking:all:20260407  → 4월 7일의 랭킹
ranking:all:20260408  → 4월 8일의 랭킹
ranking:all:20260409  → 4월 9일의 랭킹
```

"시간의 양자화(Time Quantization)"는 연속적인 시간을 일정 단위로 나누어 집계 구간을 만드는 것이다. 4월 7일의 키에는 4월 7일에 발생한 이벤트만 반영된다. 4월 8일이 되면 새로운 키가 시작되고, 모든 상품이 0점에서 출발한다.

```
4월 7일 (ranking:all:20260407)
  상품 A: 조회 200건, 주문 30건 → 200×0.1 + 30×0.7 = 41
  상품 B: 조회 500건, 주문 100건 → 500×0.1 + 100×0.7 = 120

4월 8일 (ranking:all:20260408)
  상품 A: 0에서 시작
  상품 B: 0에서 시작
  → 오늘의 실적만으로 경쟁
```

상품 B가 오늘 더 관련 있다면, 오늘의 랭킹에서 상위에 올라간다. 6개월의 누적은 관계없다. 단순하고, 설명하기 쉽고, 운영하기 편하다. "어제 랭킹에 오류가 있었다"면 해당 키만 삭제하면 된다. 가중치를 바꾸고 싶으면 내일 키부터 적용하면 된다.

### 자르면 끊어진다 — 콜드 스타트

하지만 시간을 자르면 **경계에서 단절이 생긴다.**

4월 8일 0시 0분. 새로운 키가 시작된다. 아직 아무 이벤트도 발생하지 않았다.

```
GET /api/v1/rankings?date=20260408&size=20&page=1

→ ZREVRANGE ranking:all:20260408 0 19 WITHSCORES
→ (empty list)
```

텅 빈 목록. 유저가 새벽에 "오늘의 인기 상품"을 조회하면 아무것도 없다. 어제 인기 있었던 상품이 오늘 0시에 갑자기 관련 없어지는 것은 아닌데, 시스템은 그렇게 동작한다.

이것이 **콜드 스타트(Cold Start)**다. 집계 윈도우가 시작되는 시점에 데이터가 없어서 랭킹이 의미 없어지는 구간이 발생한다. 아침이 되어서야 비로소 의미 있는 랭킹이 형성된다면, **하루 중 절반 가까운 시간 동안 랭킹이 불완전한 상태**다.

여기서 중요한 걸 인식해야 한다. 콜드 스타트는 외부 요인이 만든 문제가 아니다. **시간을 자른다는 설계 결정이 스스로 만든 문제다.** 연속적인 것을 불연속적으로 만들었으니 경계에서 단절이 생기는 것은 당연하다.

## Carry-Over — 양자화가 감쇠를 흉내 내는 순간

### 전날 점수의 잔향

콜드 스타트를 완화하기 위해, 전날 점수의 일부를 다음 날 초기 데이터로 가져온다. Score Carry-Over다.

```
전날 (ranking:all:20260407)
  상품 A: 150점
  상품 B: 80점
  상품 C: 30점

오늘 (ranking:all:20260408) — Carry-Over 10%
  상품 A: 15점 (150 × 0.1)
  상품 B: 8점  (80 × 0.1)
  상품 C: 3점  (30 × 0.1)
```

Redis에서는 `ZUNIONSTORE`로 구현한다.

```
ZUNIONSTORE ranking:all:20260408 1 ranking:all:20260407 WEIGHTS 0.1
```

전날 점수에 0.1을 곱해서 다음 날 키에 복사한다. 유저가 새벽에 조회해도 "어제 인기 있었던 상품"이라도 보인다.

### 100%면 누적, 0%면 단절

여기서 가중치 0.1이 핵심이다.

100%를 복사하면 어떻게 되는가?

```
4월 7일: 상품 A 150점
4월 8일: 150점 복사 + 오늘 이벤트 → 150 + α
4월 9일: (150 + α) 복사 + 오늘 이벤트 → 150 + α + β
```

전날 점수를 그대로 가져오면 점수가 계속 쌓인다. **키를 분리하지 않고 하나의 키에 누적하는 것과 다를 바 없다.** 시간을 자른 의미가 사라진다.

0%면? 콜드 스타트가 그대로다.

10%면?

```
4월 7일: 상품 A 150점
4월 8일: 15점 + 오늘 이벤트
4월 9일: (15 + 오늘) × 0.1 + 내일 이벤트

→ 이틀 지나면 4월 7일의 점수는 1.5점 (150 × 0.1 × 0.1)
→ 사실상 사라진다
```

**"기억은 하되, 지배당하지 않는" 균형.** 콜드 스타트는 완화하면서, 이틀만 지나면 과거의 영향은 사실상 소멸한다. 오늘의 랭킹은 오늘의 이벤트가 결정한다.

### 잠깐, 이거 어디서 본 구조 아닌가

여기서 한 발짝 물러서 보자. Carry-Over 10%가 하는 일을 다시 정리하면 이렇다.

- 하루가 지나면 점수가 10%로 줄어든다
- 이틀이 지나면 1%로 줄어든다
- 사흘이 지나면 0.1%로, 사실상 0이다

**이건 시간 감쇠(Time Decay)와 같은 효과다.** 시간이 지날수록 과거 점수의 영향력이 지수적으로 줄어든다. 다만 연속적이지 않고 하루 단위로 계단식으로 떨어질 뿐이다.

```
[시간 감쇠 - 연속]           [Carry-Over - 이산]
점수 │\                     점수 │─┐
     │  \                       │  └─┐
     │    \                     │     └─┐
     │      \___               │        └───
     └──────── 시간            └──────── 시간
```

Carry-Over는 양자화의 부작용을 메우기 위해 도입했지만, 결과적으로 **양자화가 감쇠를 흉내 내는 형태**가 된다. 시간을 자른 다음, 자른 경계를 Carry-Over로 부드럽게 잇는 것. 결국 "과거가 서서히 줄어든다"는 같은 지점에 도착한다.

그렇다면 질문이 생긴다. **처음부터 자르지 않고 녹이면 안 되는 건가?**

## 시간을 녹인다 — 감쇠

### Time Decay — 경계가 없는 접근

시간의 양자화가 "자른다"면, Time Decay는 "녹인다". 점수에 시간 함수를 곱해서, 오래된 점수일수록 영향력이 줄어들게 만드는 방식이다.

```
score = raw_score × decay(time_elapsed)

decay(t) = 1 / (1 + α × t)

예시 (α = 0.1, 시간 단위: 시간):
  1시간 전 이벤트: score × 0.91
  6시간 전 이벤트: score × 0.63
  24시간 전 이벤트: score × 0.29
  48시간 전 이벤트: score × 0.17
```

경계가 없다. 자정이 되어도 리셋되지 않는다. 점수는 시간이 지남에 따라 연속적으로 줄어든다. **콜드 스타트가 구조적으로 존재하지 않는다.**

새벽 0시 1분에 랭킹을 조회해도, 어제의 이벤트들이 감쇠된 점수로 여전히 반영되어 있다. 경계를 자르지 않았으니 메울 필요도 없다. Carry-Over가 필요 없다.

### 그런데 왜 양자화를 고르는가

Time Decay가 더 우아해 보인다. 콜드 스타트가 없고, Carry-Over도 필요 없고, 경계의 단절도 없다. 그런데 실무에서는 양자화를 고르는 경우가 많다. 왜?

**설명할 수 있어야 한다.** "오늘의 인기 상품"은 누구나 이해한다. "시간 감쇠 함수 α=0.1을 적용한 가중 점수 기준 상위 상품"은 기획자도, 판매자도, CS팀도 이해하지 못한다. "왜 이 상품이 3위인가?"라는 질문에 "오늘 주문이 많아서"는 답이 되지만, "감쇠 함수의 기울기와 이벤트 발생 시점의 상호작용으로 인해"는 답이 되지 않는다.

**디버깅할 수 있어야 한다.** 양자화된 키는 `ZREVRANGE ranking:all:20260408 0 9 WITHSCORES`로 바로 확인할 수 있다. 오늘의 랭킹은 오늘 키를 보면 된다. Time Decay는 조회 시점마다 점수가 달라지기 때문에 "지금 이 순간의 랭킹"을 재현하기 어렵다.

**운영할 수 있어야 한다.** 특정 날짜의 랭킹에 오류가 있으면 해당 키만 삭제하면 된다. 가중치를 변경하면 내일 키부터 적용하면 된다. Time Decay에서 파라미터를 변경하면, 과거의 모든 이벤트에 소급 적용할 것인지, 변경 시점부터만 적용할 것인지라는 새로운 문제가 생긴다.

| | 양자화 (자르기) | 감쇠 (녹이기) |
|---|---|---|
| 콜드 스타트 | 있음 (Carry-Over로 완화) | 없음 |
| 설명 가능성 | 높음 ("오늘 주문 많아서") | 낮음 (함수 파라미터 설명 필요) |
| 디버깅 | 쉬움 (키 하나만 보면 됨) | 어려움 (시점마다 결과 다름) |
| 운영 유연성 | 높음 (키 단위 조작 가능) | 낮음 (파라미터 변경의 소급 문제) |
| 구현 복잡도 | 낮음 (ZINCRBY + ZUNIONSTORE) | 높음 (조회마다 감쇠 계산) |

**어느 쪽이 "정답"이 아니다.** 콜드 스타트를 감수하고 단순성을 취할 것인가, 복잡성을 감수하고 연속성을 취할 것인가 — 서비스의 조건에 따라 다른 트레이드오프다.

## 어떤 방식을 골라도 남는 문제

### Matthew Effect는 스케일만 바뀐다

앞에서 "노출이 노출을 만든다"는 자기 강화 구조를 봤다. 누적의 문제라고 생각하기 쉽지만, **시간을 잘라도 이 구조는 사라지지 않는다.**

일별 키를 쓴다고 하자. 4월 8일 아침 9시, 출근한 유저들이 랭킹을 본다. 이 시점에 이미 새벽 동안 몇 건의 이벤트로 상위에 올라간 상품이 있다. 그 상품이 노출되고, 클릭되고, 구매된다. 점수가 올라간다. 오후가 되면 그 상품의 점수는 다른 상품이 따라잡기 어려운 수준이 된다.

```
6개월 누적:     상품 A (13,500) >> 상품 B (120)
일별 양자화:    상품 X (오전 85) >> 상품 Y (오전 12)
```

스케일이 6개월에서 하루로 줄었을 뿐, **먼저 노출된 상품이 더 노출되는 구조는 동일하다.** Time Decay도 마찬가지다. 감쇠 속도가 동일한 이상, 먼저 점수를 쌓은 상품이 유리한 구조는 변하지 않는다.

시간의 양자화든 감쇠든, 이 방법들이 해결하는 것은 **"과거가 현재를 지배하는 문제"**다. 하지만 **"현재 안에서의 자기 강화"**는 별도의 문제이고, 시간을 다루는 것만으로는 해결되지 않는다. 이건 랜덤 노출, 탐색/활용(Exploration/Exploitation) 전략 같은 다른 차원의 설계가 필요한 영역이다.

문제를 해결하면 더 작은 문제가 생긴다. 누적의 6개월 스케일 문제를 양자화로 1일 스케일로 줄였지만, 1일 안에서 같은 패턴이 반복된다. **완전한 해결이 아니라 축소다.** 이걸 알고 쓰는 것과 모르고 쓰는 것은 다르다.

## 마무리 — 결국 같은 질문이다

양자화와 감쇠. 이름도 다르고, 구현도 다르고, 트레이드오프도 다르다. 하지만 출발점은 같다.

> **과거는 현재보다 덜 중요해야 한다.**

이 한 문장이 랭킹 시스템의 시간 설계를 관통하는 본질이다.

누적은 과거와 현재를 동등하게 취급한다. 그래서 6개월 전의 인기가 오늘의 랭킹을 지배한다. 이걸 깨뜨리기 위해 과거의 영향력을 줄여야 하고, 줄이는 방법이 두 가지 — 자르기와 녹이기다.

> **자르기 (양자화)**는 과거를 끊는다
> - 경계가 명확하고, 설명 가능하고, 운영하기 쉽다
> - 대신 경계에서 단절이 생기고, Carry-Over로 메워야 한다
> - Carry-Over를 붙이는 순간, 결국 감쇠를 흉내 내게 된다

> **녹이기 (감쇠)**는 과거를 줄인다
> - 경계가 없으니 단절도 없고, 콜드 스타트도 없다
> - 대신 설명하기 어렵고, 디버깅하기 어렵고, 운영 파라미터를 변경하기 어렵다

> **어떤 방식을 골라도**
> - "현재 안에서의 자기 강화(Matthew Effect)"는 남는다
> - 6개월 스케일의 문제가 1일 또는 수시간 스케일로 축소될 뿐, 사라지지 않는다
> - 이건 시간을 다루는 것만으로는 해결할 수 없는 별도의 문제다

결국 랭킹의 시간 설계에서 정답은 없다. 양자화가 옳은 것도 아니고, 감쇠가 옳은 것도 아니다. **"과거의 영향력을 얼마나, 어떤 방식으로 줄일 것인가"** — 이 질문에 대한 답이 서비스의 조건과 운영 역량에 따라 달라질 뿐이다.

"시간을 자르면 공정해진다"가 아니다. **"과거를 현재보다 덜 중요하게 만들면, 지금 관련 있는 것이 보인다."** 자르든 녹이든, 본질은 그것이다.
